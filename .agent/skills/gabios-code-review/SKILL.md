> Stack: React Router v7 ┬╖ Hono ┬╖ Cloudflare Workers ┬╖ D1/Drizzle ┬╖ Better Auth ┬╖ Vercel AI SDK v6 ┬╖ R2 ┬╖ Vectorize ┬╖ Queues ┬╖ Durable Objects

## 1. Cloudflare Workers Runtime

### Bindings & Environment
- [ ] All `env` bindings typed via `Env` interface (auto-generated by `wrangler types`)
- [ ] No singleton instances that capture `env` тАФ auth, DB, AI must be **per-request**
- [ ] `ExecutionContext.waitUntil()` used for fire-and-forget work (not blocking the response)
- [ ] No Node.js-only APIs used without `nodejs_compat` flag awareness

### Worker Entry (`workers/app.ts`)
- [ ] `/api/*` routes correctly routed to Hono (except `/api/auth/*` тЖТ React Router)
- [ ] `scheduled()` handler only dispatches tasks, never does heavy compute
- [ ] `queue()` handler acks messages even on partial failure (prevent infinite retries)

```typescript
// тЭМ Singleton capturing env at module level
const db = drizzle(env.DB); // env not available here!

// тЬЕ Per-request instantiation
export const agentWorker = async (batch, env, ctx) => {
  const db = drizzle(env.DB); // тЬУ fresh per invocation
};
```

## 2. D1 Database & Drizzle ORM

### Query Safety
- [ ] All queries use **Drizzle prepared statements** (`.where(eq(...))`) тАФ never raw SQL with string interpolation
- [ ] `crypto.randomUUID()` used for all new IDs (not auto-increment, not `Date.now()`)
- [ ] Soft delete: queries filter `deletedAt IS NULL` on tables with `softDelete` mixin
- [ ] JSON columns (`config`, `details`, `metadata`, `steps`) always `JSON.parse`/`JSON.stringify` safely

### Schema Integrity
- [ ] New tables include `...timestamps` helper
- [ ] Destructive tables include `...softDelete` helper
- [ ] Foreign keys reference correct parent table (e.g., `tasks.projectId тЖТ projects.id`)
- [ ] Status fields use known enum values documented in schema comments
- [ ] Migrations generated via `npm run db:generate` тАФ never hand-written

### Performance
- [ ] No N+1: batch queries or joins used for related data (e.g., tasks + events)
- [ ] D1 row limits respected: `LIMIT` on list queries, paginated responses
- [ ] No `SELECT *` equivalent тАФ pick only needed columns for large tables

```typescript
// тЭМ N+1 pattern
for (const task of allTasks) {
  const events = await db.select().from(taskEvents).where(eq(taskEvents.taskId, task.id));
}

// тЬЕ Batch query
const events = await db.select().from(taskEvents)
  .where(inArray(taskEvents.taskId, allTasks.map(t => t.id)));
```

## 3. Hono API Routes

### Middleware Chain
- [ ] Protected routes sit **after** `authMiddleware` + `tenantMiddleware` in `server/index.ts`
- [ ] Destructive operations (DELETE, admin routes) use `requireRole('owner', 'admin')`
- [ ] Public routes (`/health`) registered **before** auth middleware

### Request Handling
- [ ] All request bodies validated with **Zod schemas** before processing
- [ ] Error responses use consistent format: `{ error: string }` with proper HTTP status
- [ ] Route handlers never swallow errors silently тАФ always log + return error response
- [ ] CORS `origin: "*"` must be locked to actual domains before production

```typescript
// тЭМ No validation
app.post("/", async (c) => {
  const body = await c.req.json(); // trusting random input
  await db.insert(agents).values(body);
});

// тЬЕ Zod-validated
const CreateAgentSchema = z.object({
  name: z.string().min(1).max(100),
  modelId: z.string().default("@cf/meta/llama-3.1-8b-instruct"),
});

app.post("/", async (c) => {
  const result = CreateAgentSchema.safeParse(await c.req.json());
  if (!result.success) return c.json({ error: result.error.flatten() }, 400);
  await db.insert(agents).values({ id: crypto.randomUUID(), ...result.data });
});
```

## 4. Authentication (Better Auth)

### Per-Request Auth
- [ ] `createAuth(env)` called **per-request** тАФ never cached as a module-level singleton
- [ ] `baseURL` correctly switches between dev (`localhost:8787`) and prod (`gabios.ness.workers.dev`)
- [ ] Session cookies: `HttpOnly`, `Secure`, `SameSite=Lax` (Better Auth defaults тАФ don't override)

### Authorization
- [ ] All mutations check user role via `c.var.user.role`
- [ ] Owner-only operations: billing, member management, API key generation
- [ ] Admin operations: agent CRUD, channel management, workflow management
- [ ] Member operations: chat, read-only dashboard

### Session Security
- [ ] No session data in localStorage or URL params
- [ ] `AUTH_SECRET` stored in `.dev.vars` (local) and Wrangler secrets (production) тАФ never in `wrangler.jsonc`

```typescript
// тЭМ Singleton auth (will cause D1 write-lock conflicts)
const auth = createAuth(globalEnv);
export default { fetch(req) { auth.api.getSession(...) } }

// тЬЕ Per-request auth
export default {
  fetch(req, env) {
    const auth = createAuth(env);
    // ...
  }
}
```

## 5. Multi-Tenant Isolation

### Database Scoping
- [ ] All queries scoped to `c.var.tenantId` or user's organization
- [ ] No cross-tenant data leakage: a user from Org A must NEVER see Org B data
- [ ] Tenant middleware resolves `tenantDb` and `tenantId` before any route handler

### R2 Storage
- [ ] Upload paths always prefixed with `tenant-{id}/`
- [ ] Download URLs are **signed** with expiration тАФ never public bucket access
- [ ] Path traversal prevented: reject keys containing `..` or absolute paths

### Vectorize
- [ ] Queries always include tenant namespace/filter
- [ ] Embeddings never mixed across tenants in the same index partition

### IDOR Prevention
- [ ] Resource access always validates ownership: `WHERE id = ? AND org_id = ?`
- [ ] Never trust client-provided IDs without server-side ownership check
- [ ] API responses never include IDs or data from other tenants

```typescript
// тЭМ IDOR vulnerability
app.get("/:id", async (c) => {
  const agent = await db.select().from(agents).where(eq(agents.id, c.req.param("id"))).get();
  return c.json(agent); // Any user can access any agent!
});

// тЬЕ Org-scoped query
app.get("/:id", async (c) => {
  const agent = await db.select().from(agents)
    .where(and(eq(agents.id, c.req.param("id")), eq(agents.orgId, c.var.tenantId)))
    .get();
  if (!agent) return c.json({ error: "Not found" }, 404);
  return c.json(agent);
});
```

## 6. AI Agent Safety (Vercel AI SDK v6)

### Prompt Injection
- [ ] System prompts are **server-side only** тАФ never constructed from user input
- [ ] `SOUL.md` content is sanitized before injection into system prompt
- [ ] User messages are clearly delimited from system context
- [ ] Agent tools never execute arbitrary code from user input

### Agent Loop
- [ ] `maxSteps` is capped (currently 3) тАФ prevent infinite tool-calling loops
- [ ] `maxLoopIterations` respected from agent config
- [ ] Every tool call and result logged to `taskEvents` for auditability
- [ ] Failed LLM calls set task status to `"failed"` тАФ not silently ignored

### Tool Safety
- [ ] `request_approval` tool correctly halts execution (sets status to `awaiting_approval`)
- [ ] `mark_task_done` sets status and logs summary тАФ never called without completion verification
- [ ] Destructive tools (delete, external API calls) always require approval flow
- [ ] Tool outputs never include raw error stack traces to the LLM

### Token Budget
- [ ] `costInTokens` tracked per task
- [ ] Department `budgetLimit` checked before dispatching new tasks
- [ ] No uncapped `maxTokens` тАФ always use agent config value (default 4096)

```typescript
// тЭМ Unsanitized user content in system prompt
const systemPrompt = `You are ${agent.name}. ${userProvidedInstructions}`;

// тЬЕ Clear separation
const systemPrompt = `You are ${agent.name}.\n\n${sanitize(agent.soulMd)}`;
const messages = [
  { role: "system", content: systemPrompt },
  { role: "user", content: userMessage } // Never mixed with system
];
```

## 7. Durable Objects & Queues

### Durable Objects (Meeting Rooms)
- [ ] WebSocket upgrade check present before accepting connection
- [ ] `state.acceptWebSocket()` used (Hibernation API) тАФ not manual `addEventListener`
- [ ] `blockConcurrencyWhile()` used in constructor for state initialization
- [ ] No unbounded state growth тАФ implement cleanup/eviction logic

### Queues (Agent Tasks)
- [ ] Messages **always** acked (`message.ack()`) тАФ even on error (prevent poison messages)
- [ ] `max_retries: 3` respected тАФ tasks that fail 3x should be marked `"failed"`
- [ ] Queue payloads are minimal (just `taskId`) тАФ full data fetched from D1
- [ ] Dead letter handling: log permanently failed tasks

```typescript
// тЭМ Missing ack on error path
catch (err) {
  console.error(err);
  // message never acked тЖТ infinite retry loop!
}

// тЬЕ Always ack
catch (err) {
  console.error("Agent Worker error:", err);
  await db.update(tasks).set({ status: "failed" }).where(eq(tasks.id, taskId));
  message.ack(); // тЖР prevent infinite retries
}
```

## 8. React Router v7 (Frontend)

### Server/Client Boundary
- [ ] `*.server.ts` files never imported from client components
- [ ] `loader`/`action` functions handle errors gracefully (return error responses, don't throw)
- [ ] Sensitive data (API keys, secrets) never returned from loaders to the client

### Route Structure
- [ ] Dashboard routes live under `dashboard/layout.tsx` with shared navigation
- [ ] Auth-required routes verify session in loader
- [ ] Loading states shown while data fetches (no blank screens)

### Component Patterns
- [ ] No `any` types тАФ use proper TypeScript interfaces
- [ ] Event handlers don't call API directly тАФ use React Router `Form` or `useFetcher`
- [ ] Client state synced with server state after mutations

### Audit Logging
- [ ] All admin actions logged to `auditLogs` table
- [ ] Log entries include: `actorId`, `action`, `targetType`, `targetId`, `details`, `ipAddress`
- [ ] Agent actions logged to `taskEvents` with `actorType: "agent" | "system" | "human"`

### Error Handling
- [ ] No `console.log` of sensitive data (tokens, passwords, API keys)
- [ ] Structured error responses with correlation IDs where possible
- [ ] Wrangler `observability.enabled: true` for production monitoring

## 10. Code Quality Standards

### TypeScript
- [ ] No `any` types тАФ use `unknown` and narrow, or define proper interfaces
- [ ] `HonoEnv` type used consistently across all Hono middleware and routes
- [ ] `Env` type from `worker-configuration.d.ts` used for all binding access

### Naming
- [ ] Files: `kebab-case.ts` for routes/middleware, `PascalCase.tsx` for components
- [ ] DB columns: `snake_case` in schema, camelCase in TypeScript
- [ ] API routes: `kebab-case` paths (`/api/agent-roles`, not `/api/agentRoles`)

### Anti-Patterns to Flag

```typescript
// тЭМ Magic strings for status
if (task.status === "3") { ... }

// тЬЕ Use the documented enum values
if (task.status === "in_progress") { ... }

// тЭМ Untyped JSON parsing
const config = JSON.parse(channel.config);

// тЬЕ Validated JSON
const config = ChannelConfigSchema.parse(JSON.parse(channel.config));

// тЭМ Mutable module-level state (Workers are shared!)
let requestCount = 0;

// тЬЕ Use Durable Objects or KV for shared state
const count = await env.KV.get("request-count");
```
